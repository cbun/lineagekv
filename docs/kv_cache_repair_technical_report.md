# Stale-Memory KV Cache Repair Technical Report

Updated: 2026-05-20

## Claim

Long-lived AI agents can preserve full memory history while neutralizing stale memories inside the model's live KV cache. Instead of deleting old memories from the prompt and rebuilding a long prefill cache, the system records memory lineage, maps stale/current records to token spans, edits stale value vectors after prefill, and decodes from the repaired cache.

The field-impact result is strongest as a systems claim:

- keep historical memory records,
- repair stale influence at runtime,
- preserve current source provenance,
- avoid expensive prompt re-prefill at long context.

## Method

The method has two phases: memory-write protocol and runtime cache repair.

### Memory-Write Protocol

1. Store every memory with `memory_key`, text, value, timestamp, status, and metadata.
2. When a newer memory is written to the same key, mark the previous current record as superseded.
3. Record an explicit lineage edge: `memory_edges(stale_id, current_id)`.
4. Do not delete superseded records. They remain available for audit and temporal reasoning.

### Runtime Repair

1. Render the full memory prefix with neutral aliases.
2. Tokenize the rendered prefix and record source-aligned token spans.
3. Prefill the model on the full prompt/cache.
4. Build a `CacheEditPlan` from memory lineage and render spans.
5. For selected layers, zero stale value vectors while preserving keys.
6. Decode from the repaired cache.
7. Repair or verify provenance from current memory IDs.

This differs from prompt deletion. Prompt deletion removes stale records before prefill. KV repair keeps the full prefix/cache and edits the stale value state after prefill.

| Qwen2.5-0.5B | full cache | 0.362 | 0.450 | 0.650 |
| Qwen2.5-0.5B | value-only | 0.412 | 0.650 | 0.375 |

Interpretation: stale values carry harmful obsolete content; stale keys preserve useful address/evidence structure. Deleting keys suppresses stale use but damages evidence routing.

## Limitations

1. The strongest 7B result is MLX 4-bit Qwen2.5, not full-precision CUDA.
2. The direct memory-store-to-MLX integration is one simple case; it proves the path, not a large deployed-agent study.
3. Exact cache-localization policies are model- and layer-dependent.
4. Text+timestamp targeting assumes usable timestamps and sufficiently explicit memory text.
5. The current benchmark is generated and should be released with clear scope rather than sold as a natural production log.
6. Second-family evidence is mixed: TinyLlama and SmolLM2 are useful boundaries, not clean full replications.
